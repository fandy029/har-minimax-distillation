# 第2章 数据预处理和Prompt设计

## 2.1 数据集的选取

### 2.1.1 选取标准

HAR 系统首先要靠传感器捕捉人体运动数据。目前常用的传感器有四种：惯性测量单元（IMU）、表面肌电传感器、足底压力传感器和视觉传感器。其中 IMU 出场率最高，因为它体积小、功耗低、成本可控，还能直接集成到手表、手机、腰带上这类常见穿戴物里。IMU 一般把三轴加速度计和三轴陀螺仪集成在一起——加速度计测量身体各轴向的加速度变化，反映出运动强度和方向的改变；陀螺仪测量各轴向的角速度，能捕捉到姿态旋转。人做不同类型动作时，这些信号在时域和频域上表现出的特征模式有明显差别，这也是后续识别的基础。

IMU 的佩戴位置直接影响采集到的信号特征。手腕上的传感器主要反映手臂运动模式，像抬手、绕腕这类动作很清楚；腰部位置贴近身体重心，更适合识别行走、跑步这类全身性活动；脚踝对步态的细微变化更敏感；大腿和背部则能同时提供上半身和下半身的运动信息。所以到底贴在哪儿，得看要识别的动作类型。

采样频率也有讲究。人体日常运动产生的信号频率成分基本落在 0.5Hz 到 20Hz 之间。根据奈奎斯特定理，采样率至少得是信号最高频率的两倍才能完整保留原始信息。实际应用中一般设在 20Hz 到 100Hz，这样既能保住有效频率成分，又不会产生过大的数据量。

基于上面这些考虑，本次挑数据集时主要把握了五个原则：

第一，传感器类型要多样。不光包括专门的 IMU，智能手机自带的加速度计也纳入进来，让后续模型能适应不同设备采到的信号。

第二，佩戴位置要丰富。手部、腰部、背部、大腿都覆盖到，上肢、下肢和躯干的动作特征都有了代表。

第三，活动类别的复杂度拉开层次。从最简单的 4 类逐步扩展到 18 类，既照顾了二分类场景，也覆盖了多分类需求，日常起居和运动健身的动作都有涉及。

第四，采样率有梯度。从 20Hz 到 100Hz 都有，低频采样考验算法鲁棒性，高频采样保留更多运动细节。

第五，数据形态上同时提供原始时域信号和预提取特征。原始信号方便开发者自己做特征工程，预提取特征可以直接用于基准对比。

**PAMAP2**（Physical Activity Monitoring for Aging People）由德国亚琛工业大学采集，面向老年人身体活动监测。数据通过 IMU 以 100Hz 采样率采集手部 6 轴信号（3 轴加速度+3 轴陀螺仪）。包含 5 类活动：lying（躺）、sitting（坐）、standing（站）、walking（走）、jogging（跑）。该数据集的特点是手部重力投影对静态姿势的区分效果很明显——手腕位置一变，重力在三轴上的分量就跟着换了。动态活动则主要靠频率特征来区分：走的支配频率在 0.8Hz 左右，跑则在 1.5Hz 以上。

**KuHar**（Korean Human Activity Recognition）由韩国梨花女子大学采集，传感器佩戴在腰部，采样率 100Hz。8 通道数据来自加速度计和陀螺仪。18 类活动是 5 个数据集中最多的，涵盖静态姿势（站立、坐下、躺着）、过渡动作（坐-站、躺-站）、日常活动（走路、后退、转圈）和体育运动（跑、跳、深蹲、俯卧撑、乒乓球）。类别多加上一些类别之间特征重叠（比如三种走路变体：正常走、后退、转圈），让这个数据集成为验证蒸馏方法效果的最好场景。

**HARTH**（Human Activity Recognition Translational Healthcare）使用背部加大腿的双 IMU 配置，采样率 50Hz，经 10Hz 低通滤波预处理。6 通道数据分别来自两个位置的加速度计。8 类活动为：walk（走）、run（跑）、shuffle（慢走）、stairs_up（上楼）、stairs_down（下楼）、stand（站）、sit（坐）、lying（躺）。双传感器配置对静态姿势的区分非常有效——坐着时大腿加速度的 Z 轴重力分量（thigh_gz）接近 0.95，站着时约为 -0.16，躺着时约为 0.21，这三个值拉开得很开。但该数据集类别严重不平衡：sit 类有 31,737 个训练窗口，占了总量的一半左右，而 run 只有 3,185 个窗口。这对蒸馏方法处理不平衡问题的能力是个实际考验。

**MotionSense** 使用 iPhone 6s 腰部加速度计采集，采样率 20Hz。数据采用去重力后的 userAcceleration（只保留运动线性加速度，不含重力分量），通道数 3。6 类活动为：sit（坐）、stand（站）、walk（走）、jog（慢跑）、upstairs（上楼）、downstairs（下楼）。去掉重力后，静态姿势（坐和站）的加速度都接近零向量，它们的区分转而依赖水平面加速度幅值的细微差异。上楼和下楼的识别则主要靠垂直方向加速度的方向判断。

**Gait** 是最小的数据集，只有 3 轴加速度计数据，采样率 40Hz。4 类活动为：sit_on_bed（坐在床上）、sit_on_chair（坐在椅子上）、lying（躺着）、ambulating（行走）。总共 737 个训练窗口，类别严重不平衡——lying 有 478 个窗口，其他 3 类加起来才 259 个。静态姿势的区分靠重力方向：躺着时重力方向接近水平，坐着时接近垂直向下。动态的判断依赖运动强度和步态峰值数。

### 2.1.3 数据预处理方法

人体运动是连续时序过程，需要把连续信号切分为独立的分析窗口后才能做分类。滑动窗口分割是这一步的核心方法：用一个固定长度的窗口在信号序列上按固定步长滑动，每个窗口位置对应一个独立样本。

窗口长度的选择直接影响识别效果。太短则单个窗口包含的运动信息不够，分不清活动类型；太长则窗口内可能混进多种活动的信号，边界变模糊。根据人体运动特点和传感器采样频率，常用窗口长度在 1 秒到 3 秒之间。本文对 5 个数据集统一采用窗口大小 128 步、步进 64 步的配置，重叠率 50%。这样设计有几个考虑：128 个采样点在 20Hz 下覆盖 6.4 秒、在 100Hz 下覆盖 1.28 秒，都能包含至少一个完整的运动周期；步进 64 步在保证样本多样性的同时避免过度冗余；另外窗口长度 128（2 的 7 次方）和 CNN 的卷积与池化设计天然兼容。

50% 重叠率的做法是：窗口从序列起点开始，每次前进 64 步取 128 步作为下一个样本。这种做法不会丢失活动转换处的重要信息，相比不重叠的切分方式能生成大约一倍的样本量。

所有数据集在窗口划分后，去掉包含 NaN 值的窗口，每个窗口的标签由窗口起始点的标签决定。之后按 85%/15% 做分层采样，分出训练集和测试集；训练集里再分出约 15% 作为验证集。分层采样保证了各类别在三个子集中的比例一致。

### 2.1.4 特征提取

特征提取的目的是从原始传感器时序数据中算出有物理含义的数值，供 prompt 构建时使用。本文对每种数据集都设计了一套专属的特征计算函数，特征类型可以分为三类：

（1）时域统计特征。直接从信号的时间序列计算得到，反映信号的统计特性。包括：各通道均值、标准差（反映运动幅度）、信号能量（加速度幅值平方和的均值）、急动度 jerk（加速度差分量的均方根，反映运动的剧烈程度变化）、冲击性 impulsiveness（信号峰值与均值之比，反映是否有瞬时剧烈运动）、峰值计数（超过动态阈值的局部极大值个数）。

（2）频域特征。通过傅里叶变换（FFT）将信号从时域转到频域分析。主要包括：支配频率（功率谱最大的频率分量，对走和跑的区分很有用——走约 0.8Hz，跑约 1.5Hz 以上），以及频谱的能量分布情况。

（3）重力方向特征。加速度计采集的信号里天然包含重力加速度分量（约 9.8m/s²）。通过对窗口做 10Hz 低通滤波可以近似提取重力向量在三个轴上的投影方向，这是区分静态姿势的关键。以 HARTH 的大腿传感器为例：坐着时重力方向基本沿着传感器 Z 轴（thigh_gz ≈ 0.95），站着时重力方向接近传感器 Y 轴（thigh_gz ≈ -0.16），躺着时在中间（thigh_gz ≈ 0.21）。去掉重力后的自由加速度则可以更纯粹地反映运动分量。

以 Gait 数据集为例的特征计算：

```
# 10Hz 低通滤波提取重力方向
gfr = uniform_filter1d(window[:, 0], 10, axis=0).mean()  # 前后方向
gve = uniform_filter1d(window[:, 1], 10, axis=0).mean()  # 上下方向
gla = uniform_filter1d(window[:, 2], 10, axis=0).mean()  # 左右方向

# 去掉重力后的自由加速度
free = window - [gfr, gve, gla]
free_mag_std = sqrt((free**2).sum(axis=1)).std()  # 自由加速度幅值标准差
peaks = count_peaks(free_mag)  # 峰值个数
```

不同数据集的具体特征集合略有差异：PAMAP2 侧重 jerk 和支配频率来区分动与静、走与跑；KuHar 额外关注加速度 Y 轴标准差和陀螺仪 X 轴标准差，因为腰部转动是区分 18 类动作的重要线索；HARTH 的核心特征是大腿加速度标准差（thigh_std）和重力方向分量（thigh_gz），因为双传感器对姿态的判断更精确；MotionSense 用去重力加速度的水平方差来区分坐和站；Gait 则完全依赖重力方向——躺着时 gve（上下方向重力分量）接近 0，坐着时接近 -1。

## 2.2 软标签生成架构

### 2.2.1 基本原理

传统分类任务中，标签是 one-hot 形式的硬标签，只反映样本属于唯一的一个类别，忽略了类别之间的相似性。拿 HAR 来说，站着和坐着在特征空间里挨得很近，但站着和跑步隔得很远——硬标签表达不出这个信息。

知识蒸馏通过引入温度参数 T 对教师模型的输出做软化处理。软化后的概率分布不像 one-hot 那样尖锐，次要类别的概率被放大，类别间的相似性关系就体现出来了。具体来讲，温度 T 调节 softmax 函数的"软硬"程度：

$$q_i = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}$$

T=1 退化为标准 softmax，概率集中在最大值；T 增大时分布变平坦，各类别之间的相对差异被拉近；T 趋于无穷时退化为均匀分布。

在本文方法中，教师不是另一个神经网络，而是 MiniMax M2.7 大语言模型。每轮调用时，我们把一个 IMU 传感器窗口计算出的物理特征写成 prompt，让大模型基于它对力学和运动学原理的理解去推理，输出 18 个类（以 KuHar 为例）各自的概率。这样做的好处是不需要预先训练一个教师网络——零样本推理就能拿到软标签。当然代价是每个窗口都要调一次 API，对时间和 API 配额有要求。

蒸馏损失函数是学生模型输出和软标签之间的 KL 散度，加上学生模型和真实标签之间的交叉熵损失，通过 α 参数加权：

$$\mathcal{L} = \alpha \cdot CE(hard) + (1-\alpha) \cdot T^2 \cdot KL(soft \parallel logits/T)$$

其中 T² 是温度补偿因子，因为 KL 散度的梯度在高温下会缩小约 1/T 倍，乘上 T² 后不同温度下的损失量级可以对齐。

### 2.2.2 按类并行生成架构

5 个数据集的训练窗口总量达到 15.5 万个，如果每个窗口单独串行调 API，时间不可接受。为此设计了一套按类并行生成架构，由四个组件串联起来：

（1）API 配置集中化。所有数据集共享同一个 api_config.py，包含 API 密钥、模型地址（MiniMax M2.7）、温度参数（0.8）、最大 token 数（10,000）、超时时间（120 秒）、请求间隔（0.3 秒）等。这样避免了每个数据集都要重复定义。

（2）数据准备阶段（prepare 脚本）。每个数据集运行一次 prepare 脚本，完成原始数据读取、标签映射、滑动窗口分割（窗口 128、步进 64）、训练/验证/测试集划分。划分后按类别拆开，每个类别的窗口数据和索引单独存成 windows.npy 和 indices.npy 文件，放在 output/per_class/class_N/ 目录下。

（3）按类生成阶段（gen 脚本）。每个类别独立启动一个生成进程，进程间通过文件锁（fcntl.LOCK_EX）互斥，保证同一个类别不会被两个进程同时处理。进程支持断点续传：每处理 5 个窗口保存一次 checkpoint，记录已完成窗口的全局索引，下次启动时自动跳过。

每个窗口的生成流程为：计算当前窗口的物理特征 → 构建 prompt 并调用大模型 API → 解析返回的 JSON 概率向量 → 将软标签写入全局 soft_all.npy 数组。如果 API 返回无法解析，该窗口降级为 one-hot 硬标签。

（4）结果合并阶段（merge 脚本）。所有类都生成完后，merge 脚本遍历 train_labels.npy 中的每个样本，按其真实类别找到对应类别的 soft_all.npy。若该样本已有有效的软标签（概率向量和大于 0），则予以保留；否则使用 one-hot 硬标签填充。合并后的软标签文件作为蒸馏训练中软标签监督的来源。

### 2.2.3 Prompt 设计原则

虽然 5 个数据集的活动类别、传感器配置、复杂度各不相同，但 prompt 设计遵循了统一的原则。

**原则一：提供数据驱动的中位数参考值。** 不是用文字描述"走路的加速度大约在 0.5 到 0.8 之间"这种模糊的范围，而是从每个类别的实际数据分布中算出中位数（p50）特征值，直接写在 prompt 里。比如 KuHar 的 prompt 中，走路类（class 11）的参考值是："energy=21, z_grav=0.57, acc_y_std=1.89, gyro_x_std=0.49"。这样大模型不需要凭经验猜测各类别的特征范围，而是直接把当前窗口的计算值和各类别的真实中心值做数值级比较。

**原则二：分层决策树结构。** 不是让模型在 18 个类别中一次性选最像的，而是先粗分后细分。以 KuHar 为例：先按能量大小分到四个层级——静态层（能量 < 0.1，含 Stand/Sit/Talk-sit/Lay 共 4 类）、过渡层（能量 0.1-5，含 Talk-stand/Stand-sit/Lay-stand/Pick/Sit-up 共 5 类）、动态层（能量 5-50，含 Push-up/Walk/Walk-back/Walk-circle/Stairs-up/Stairs-down/Ping-pong 共 7 类）、高强度层（能量 > 100，含 Jump/Run 共 2 类）。在每个层级内部再做细粒度比较。这样把 18 类的比较空间压缩到了一个合理的范围。

**原则三：基于物理规律的特征选择。** 每个数据集选取的核心特征都跟动作的物理本质挂钩。PAMAP2 用手部 jerk 和加速度 X 轴均值来区分动与静、躺与坐站；HARTH 靠大腿加速度标准差（thigh_std）分群，靠大腿重力 Z 轴分量（thigh_gz）在静态姿势内部细分；MotionSense 靠去重力加速度水平标准差来区分坐和站的微晃差异；Gait 完全依赖重力方向。这些特征选择不是拍脑袋决定的，而是从实际数据中观察各类别的特征分布得出来的。

**原则四：禁止 one-hot 输出。** Prompt 要求输出 JSON 格式的概率分布而不是单一边界判别，API 参数中 temperature=0.8 也服务于这个目标。较高温度使输出的概率不那么尖锐，保留类别间的相似性信息，为蒸馏提供有效的学习信号。

### 2.2.4 软标签质量分析

软标签的质量直接影响蒸馏效果。本文用两个维度评估质量：准确率（预测类别与真实标签一致的比例）和置信度分布（非零软标签的平均最大概率）。

|--------|---------|-------------|-----------|
| Gait | 100% | 91.2% | 0.61 |
| MotionSense | 100% | 81.6% | 0.34 |
| PAMAP2 | 86.7% | 87.3% | 0.79 |

注：准确率和置信度基于已完成部分统计，全部完成后会有完整数据。

Gait 准确率最高，有四个原因叠加在一起：类别数最少（4 类）、lying（躺着）与其他姿势的重力方向完全相反（上下分量接近 0 vs 接近 -1）、lying 的 478 个样本占了训练集 65%，绝大部分都判断对了，成了整体准确率的锚点。PAMAP2 的 5 类活动在物理特征上天然可分——动态的走和跑有明显的周期性，静态的躺/坐/站可以通过重力方向精确区分。

HARTH 的准确率最低。虽然双 IMU 配置信息丰富，但 sit（坐着）类占了近一半窗口，容易的样本多、难的样本被淹没了；而且 stand（站立）和 shuffle（慢走）在大腿传感器上特征重叠严重——两者都是轻度活动、thigh_std 都很低（0.01 vs 0.09）。KuHar 虽然类别最多（18 类），但四层级能量匹配策略把比较空间从 18 类压缩到了每个层级内的 4-7 类，让高能量动作（跳、跑）的准确率保持在较好水平（81.9% 以上），难度主要出在走路变体（正常/后退/转圈）之间，它们本身物理特征就很接近。

整体来看，5 个数据集的软标签准确率在 78% 到 91% 之间，都处于可接受的蒸馏质量区间。值得注意的是，即使大模型把某个样本判错了（比如把站立判成慢走），输出的软标签里两者往往都给了较高的概率——这恰好传递了"这两个类别确实很像"的信息，对学生模型建立更平滑的决策边界是有帮助的。

## 2.3 本章小结

本章围绕 5 个 HAR 数据集的选取、预处理、软标签生成架构、prompt 设计以及质量分析展开。

数据集选取上，5 个数据集在传感器类型、佩戴位置、活动复杂度、数据规模四个维度形成了互补。PAMAP2 侧重手部数据，KuHar 覆盖最多的活动类别，HARTH 提供双传感器配置，MotionSense 使用消费级设备，Gait 作为小数据集代表。这样的组合能更全面地考察方法在不同场景下的表现。

预处理方面，5 个数据集统一使用窗口大小 128、步进 64、50% 重叠的滑动窗口分割。部分数据集额外做了 10Hz 低通滤波来提取重力分量。每个数据集都设计了专属的特征计算函数，计算与运动物理本质相关的时域、频域和重力方向特征。

软标签生成采用按类并行的架构，支持断点续传。prompt 设计围绕四个原则：提供中位数参考值做数值级比较、分层决策树结构压缩比较空间、基于物理规律选择核心特征、禁止 one-hot 输出确保软标签信息量。最终输出合并后的软标签文件供蒸馏训练使用。

从已完成部分的评估来看，软标签准确率在 78% 到 91% 之间，处于可用范围。准确率差异主要来自数据集本身的难度——类别越少、物理特征分离度越高的数据集，软标签质量越好。
