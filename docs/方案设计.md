# 面向非平整地形的地面机器人自主导航算法研究 — 技术方案

> 本文档为毕业论文章节三「系统设计」的技术路线与部署方案

---

## 一、论文核心需求梳理

### 1.1 研究目标
实现复杂地形环境下兼具**实时性、鲁棒性与平稳性**的路径规划功能，核心技术环节：

```
地形感知 → 特征提取 → 可通行性分析 → 全局路径规划 → 局部轨迹优化
```

### 1.2 三大核心环节

| 环节 | 输入 | 输出 | 对应论文章节 |
|------|------|------|-------------|
| ① 地形特征感知 | 3D 点云 / 深度图像 | 平面拟合结果、高程地图 | 3.2 地形感知与建模 |
| ② 可通行性分析 | 高程/坡度/粗糙度地图 | 可通行区域代价地图 | 3.3 地形可通行性评估 |
| ③ 路径规划与优化 | 代价地图 | 安全、平滑、能耗优化的路径 | 3.4 路径规划算法 |

### 1.3 性能指标要求
- 点云处理帧率：≥ 10 Hz
- 全局路径规划：≤ 1s（100m × 100m 范围）
- 局部轨迹更新：≥ 20 Hz
- 支持地形坡度：0° ~ 45°
- 支持台阶高度：0 ~ 0.3m

## 二、项目组合架构

采用**三个开源项目组合**，各司其职：

```
┌─────────────────────────────────────────────────────────────┐
│                      系统架构图                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [3D 传感器]  →  ori-drs/plane_seg                          │
│  (Lidar/      │  ├── 功能：平面拟合、地面分割、边缘检测        │
│   DepthCam)   │  ├── 输出：平面参数、地面点云、高程种子点       │
│               │  └── 特点：RANSAC + 鲁棒估计，支持 ROS1/2    │
│               ↓                                              │
│  ANYbotics/grid_map                                          │
│  ├── 功能：多层栅格地图（高程/坡度/粗糙度/可通行性）           │
│  ├── 输入：平面拟合结果 + 原始点云                            │
│  ├── 输出：elevation / slope / roughness / traversability     │
│  └── 特点：通用 C++ 库，ROS2 Jazzy 兼容                     │
│               ↓                                              │
│  dfki-ric/ugv_nav4d  或  spuetz/mesh_navigation             │
│  ├── 功能：4D 路径规划 / 三角网格路径规划                     │
│  ├── 输入：可通行性代价地图                                   │
│  ├── 输出：X,Y,Z,Theta 轨迹点序列                           │
│  └── 特点：模块化、支持 Docker 部署                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 各项目职责

| 项目 | 职责 | 编程语言 | 依赖 |
|------|------|----------|------|
| `ori-drs/plane_seg` | 地形感知：平面拟合、地面分割 | C++ | PCL ≥ 1.9, ROS2 |
| `ANYbotics/grid_map` | 地图构建：多层地形特征地图 | C++ | Eigen3, ROS2 |
| `dfki-ric/ugv_nav4d` | 路径规划：4D (X,Y,Z,Theta) 规划 | C++ / Python | ROS2, OMPL, Docker |

### 2.2 数据流

```
传感器点云
    │
    ▼
┌──────────────┐    平面参数    ┌─────────────┐   多层地图    ┌──────────────┐
│  plane_seg   │ ────────────→ │  grid_map   │ ────────────→ │   ugv_nav4d  │
│  平面拟合    │   地面点云     │  地形特征    │   代价地图    │   4D 路径    │
└──────────────┘               └─────────────┘              └──────────────┘
```

## 三、详细模块设计

### 3.1 地形感知模块 — ori-drs/plane_seg

**核心算法：**
- 基于 RANSAC 的鲁棒平面拟合
- 法向量聚类 + 平面区域合并
- 边缘检测（坡度突变处标记为障碍）

**关键类/函数：**
```
PlaneFitting  — 主拟合器
  fitPlanes()     — 输入点云，输出平面列表
  getGroundPoints() — 提取地面点子集
  getPlaneParams()  — 返回平面方程 (a,b,c,d)

EdgeDetector  — 边缘检测
  detectEdges()   — 基于法向量突变检测障碍边缘
```

**适配改造点：**
- 论文创新：加入时序滤波（前后帧平面跟踪），提升不平整地形的拟合稳定性
- 边缘检测结合坡度阈值，区分"可通行坡面"与"不可通行台阶"

### 3.2 地形特征地图模块 — ANYbotics/grid_map

**地图层次设计（对应 grid_map 的 Layer 类型）：**

| Layer 名称 | 计算方式 | 用途 |
|-----------|----------|------|
| `elevation` | 平面拟合高程值直接写入 | 基础高程 |
| `slope` | elevation 的梯度计算 | 坡度评估 |
| `roughness` | 邻域高程方差 | 地面平整度 |
| `traversability` | 综合坡度+粗糙度+台阶高 | 可通行性代价 |
| `upper_bound` | 射线投射未观测区域上限 | 避障规划 |

**grid_map 核心参数：**
```yaml
resolution: 0.05        # 5cm 栅格精度
length_x: 20.0          # 20m × 20m 局部地图
length_y: 20.0
position: [0, 0, 0]     # 以机器人为中心
frame_id: "odom"
```

**适配改造点：**
- 论文创新：可通行性权重自定义（论文提出的多约束融合权重）
- 加入语义层（草地/泥土/岩石分类，如使用深度学习）

### 3.3 路径规划模块 — dfki-ric/ugv_nav4d

**规划算法：**
- 全局规划：基于采样的 RRT* 或 A*（4D 状态空间）
- 局部规划：MPC 跟踪 + 动态避障重规划

**ugv_nav4d 核心流程：**
```
user_config.yaml → 动态加载配置
    │
    ├── robot_type: "wheeled_ugv"
    ├── planner_algorithm: "rrt_star" | "a_star" | "mmp"
    ├── state_space: [x, y, z, theta]
    └── cost_weights: [slope, roughness, distance]
    │
    ▼
main.sh → 启动 ROS2 节点
    │
    ├── ugv_nav4d_node (规划器)
    ├── rviz2 (可视化)
    └── 可选：gazebo_ros (仿真)
```

**适配改造点：**
- 论文创新：替换/优化 ugv_nav4d 的启发式代价函数，使用 grid_map 的 traversability 层作为代价输入
- 轨迹平滑：加入论文提出的 B-Spline 或 MINCO 多项式轨迹优化

## 四、部署方案

### 4.1 环境要求

| 环境 | 要求 | 当前服务器 | 差距 |
|------|------|-----------|------|
| 系统 | Ubuntu 20.04/22.04/24.04 | Ubuntu 24.04 ✅ | — |
| ROS | ROS2 Humble/Jazzy | ROS2 Jazzy ✅ | — |
| Python | ≥ 3.8 | 3.12 ✅ | — |
| 内存 | ≥ 8GB | 3.7GB ❌ | 需 Docker 内存限制 |
| 磁盘 | ≥ 15GB 可用 | 7.8GB ❌ | 需清理 |
| GPU | 推荐 NVIDIA (非必需) | 无 | 无 GPU 情况下训练不可行，推理可跑 |

> ⚠️ **当前服务器内存严重不足（3.7GB）**，建议所有组件通过 **Docker** 部署并限制容器内存上限，避免与宿主机 thesis 环境冲突。

### 4.2 Docker 部署架构

```
宿主机 (Ubuntu 24.04)
│
├── Docker 容器 A: ugv_nav4d (4D 规划器)
│   ├── 镜像：ubuntu:22.04 + ROS2 Jazzy
│   ├── 内存上限：1.5GB
│   └── 端口：仅 ROS2 DDS 通信
│
├── Docker 容器 B: 地形感知 + grid_map
│   ├── 镜像：ubuntu:22.04 + PCL + grid_map
│   ├── 内存上限：1GB
│   └── 与容器 A 通过 ROS2 Topic 通信
│
└── 宿主机 ROS2 环境 (grid_map 核心库)
    └── 仅编译用，运行时放容器
```

### 4.3 部署步骤

**Step 1：准备 ROS2 工作空间**
```bash
mkdir -p ~/ros2_nav_ws/src
cd ~/ros2_nav_ws
source /opt/ros/jazzy/setup.sh
```

**Step 2：编译 plane_seg**
```bash
# ori-drs/plane_seg — 需要适配 ROS2
git clone https://github.com/ori-drs/plane_seg.git src/plane_seg
cd src/plane_seg
git checkout ros2_dev  # 切换到 ros2 分支（如有）
# 如无 ros2 分支，需手动修改 CMakeLists.txt 和头文件适配 ROS2
cd ../..
colcon build --packages-select plane_seg
```

**Step 3：安装 grid_map**
```bash
# 方式 A：apt 安装（推荐）
apt install ros-jazzy-grid-map

# 方式 B：源码编译
git clone https://github.com/ANYbotics/grid_map.git src/grid_map
colcon build --packages-select grid_map
```

**Step 4：部署 ugv_nav4d（Docker）**
```bash
# 进入 ugv_nav4d 项目目录
cd ~/path/to/ugv_nav4d

# 构建 Docker 镜像
docker build -t ugv_nav4d:jazzy .

# 运行容器
docker run --rm \
  --network host \
  --memory=1.5g \
  ugv_nav4d:jazzy \
  bash main.sh
```

**Step 5：数据集准备（验证用）**
- 使用任意点云数据集（SemanticKITTI / KITTI / 自采数据）
- 或使用 ugv_nav4d 自带的模拟地形

## 五、资源估算

### 5.1 磁盘占用

| 组件 | 源码大小 | 编译后 | Docker 镜像 | 说明 |
|------|---------|--------|-------------|------|
| plane_seg | ~5 MB | ~20 MB | ~800 MB | 含 PCL 依赖 |
| grid_map | ~20 MB | ~50 MB | ~600 MB | 含 Eigen3 |
| ugv_nav4d | ~50 MB | ~100 MB | ~2 GB | 含 OMPL + 全量依赖 |
| ROS2 Jazzy (apt) | — | — | ~4 GB (如容器内安装) | |
| 训练数据集 | — | — | — | 论文数据集独立存放 |
| **总计** | ~75 MB | ~170 MB | **~8 GB** | **容器内总计** |

> ⚠️ 当前服务器可用磁盘 **7.8GB**，非常紧张。建议：
> - 清理 thesis/datasets 中不常使用的数据（当前 4GB）
> - 或只保留论文验证用的小规模点云样本
> - 编译产物不保留，调试完即删

### 5.2 内存占用（运行时峰值）

| 进程 | 空闲 | 正常规划 | 峰值 | 持续时间 |
|------|------|---------|------|---------|
| plane_seg | ~50 MB | ~150 MB | ~300 MB | < 1s |
| grid_map | ~30 MB | ~80 MB | ~150 MB | 持续 |
| ugv_nav4d (RRT*) | ~100 MB | ~300 MB | ~600 MB | 1-3s |
| **合计** | ~180 MB | ~530 MB | **~1 GB** | — |

> 当前服务器物理内存 3.7GB，扣除系统基础占用 ~1GB，剩余 ~2.7GB 可用。
> 上述组件**合计峰值 1GB**，在无其他进程干扰时可运行，但**非常紧张**。
> **建议**：将 ugv_nav4d 和 grid_map 分容器部署，各限制 512MB。

### 5.3 CPU 占用

| 组件 | 正常 | 峰值 | 核数需求 |
|------|------|------|---------|
| plane_seg | 5-10% | 30% | 单核即可 |
| grid_map | 3-5% | 15% | 单核 |
| ugv_nav4d | 10-20% | 80% | 多核 (OMPL 采样规划) |
| **合计** | ~20% | **~100%** | 4核+ 推荐 |

> 服务器 CPU 为 4 核水平，单一规划任务可达满载，但其他进程并发时会出现卡顿。
> 建议：规划时关闭 thesis 训练进程，避免 CPU 争抢。

### 5.4 网络

- 仅开发/下载期间占用，不影响运行时
- Docker 镜像下载：ugv_nav4d ~2GB，grid_map/plane_seg ~1GB

## 六、部署过程资源占用详情

### 6.1 各阶段资源峰值

| 阶段 | 操作 | CPU 峰值 | 内存峰值 | 磁盘写入 | 时长 |
|------|------|---------|---------|---------|------|
| 下载源码 | git clone × 3 | < 5% | < 100 MB | ~150 MB | ~5 min |
| 安装依赖 | apt install / pip install | 20-50% | 300-500 MB | ~500 MB | 10-30 min |
| CMake 编译 | colcon build | **80-100%** | **800MB-1.2GB** | ~200 MB | 20-60 min |
| Docker 构建 | docker build | 50-80% | 1-1.5 GB | **2-4 GB** | 30-60 min |
| 运行时 | 正常导航 | 20-40% | ~530 MB | < 10 MB | 持续 |

### 6.2 编译资源占用（重点）

**colcon build 是资源消耗最高的环节**，原因：
- grid_map 模板元编程编译时间较长
- ugv_nav4d 依赖 OMPL，编译单元多
- 并行编译 (-j4) 会同时吃 CPU 和 内存

**建议**：
```bash
# 限制并行数，降低内存峰值
colcon build --parallel-workers 2

# 分包依次编译
colcon build --packages-select grid_map_core
colcon build --packages-select grid_map_filters
colcon build --packages-select grid_map
colcon build --packages-select plane_seg
```

### 6.3 Docker 构建资源占用

Docker 构建过程中：
- 内存峰值出现在 `colcon build` 阶段
- 磁盘写入为 Docker 层缓存，总计 ~4GB
- 构建完成后清理中间层可释放 ~1-2GB

## 七、实施计划

### 7.1 阶段划分

```
第一阶段：环境准备（1-2 天）
  ├── 清理磁盘（转移/删除 thesis/datasets 不常用数据）
  ├── 安装 ROS2 依赖包
  └── 准备 Docker 环境
  │
第二阶段：地形感知模块部署（2-3 天）
  ├── 编译安装 grid_map
  ├── 编译安装 plane_seg
  ├── 点云输入输出联调
  └── 平面拟合效果验证
  │
第三阶段：地图构建与可通行性分析（2-3 天）
  ├── 多层 grid_map 配置
  ├── traversability 计算参数调优
  └── 地形特征可视化 (rviz2)
  │
第四阶段：路径规划模块部署（3-5 天）
  ├── ugv_nav4d Docker 部署

  ├── grid_map → ugv_nav4d 接口对接
  └── 端到端联调
  │
第五阶段：论文验证实验（持续）
  ├── 多地形数据集测试
  ├── 算法性能对比
  └── 论文图表制作
```

### 7.2 每日资源预留

| 时段 | 建议活动 | 资源预留 |
|------|---------|---------|
| 白天 | 论文写作 / 阅读 | 不占用计算资源 |

## 八、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| 磁盘空间不足编译失败 | 高 | 高 | 先清理数据集；用 Docker 而非全量 apt |
| 内存不足 OOM | 高 | 高 | 全程 Docker 限制内存；分步编译 |
| plane_seg 无 ROS2 分支 | 中 | 中 | 手动适配 CMakeLists，或用 ROS1 容器桥接 |
| ugv_nav4d 无人维护，编译失败 | 中 | 中 | 切换为 mesh_navigation（更活跃） |
| 无 GPU 深度学习地形分类 | 低 | 低 | 用规则-based 可通行性分析替代 |

## 九、备选方案

如果 `ugv_nav4d` 部署困难，可切换为：

### 备选：spuetz/mesh_navigation

| 对比项 | ugv_nav4d | mesh_navigation |
|--------|-----------|-----------------|
| 规划维度 | 4D (X,Y,Z,Theta) | 3D (X,Y,Z) + 方向 |
| 地图类型 | MLS 栅格地图 | 三角网格 mesh |
| 规划算法 | RRT*/A*/MMP | Dijkstra/WaveFront |
| Docker 支持 | 有 | 有 |
| 活跃度 | 一般 | 较活跃 |
| 推荐度 | ⭐⭐⭐ | ⭐⭐⭐⭐ |

**mesh_navigation 更轻量**，适合资源受限环境，且与 plane_seg + grid_map 组合更自然（三者都基于 grid_map 生态）。

## 十、总结

| 项目 | 作用 | 资源占用 |
|------|------|---------|
| ori-drs/plane_seg | 地形感知（平面拟合） | 轻量 |
| ANYbotics/grid_map | 地图构建（多层地形特征） | 中等 |
| dfki-ric/ugv_nav4d | 路径规划（4D 规划） | 较重 / 可 Docker |
| **合计（运行时）** | — | **~1GB RAM, 单核 CPU** |
| **合计（编译时峰值）** | — | **~1.2GB RAM, 4核满载** |

> **核心建议**：全程使用 **Docker** 部署 ugv_nav4d，宿主机仅编译 grid_map/plane_seg。磁盘紧张是最大瓶颈，建议将 thesis/datasets 压缩转移后再开始部署。
